# POE2 国际服实时查价 AstrBot 插件设计

日期：2026-06-18

## 目标

开发一个用于 Path of Exile 2 国际服实时查价的 AstrBot 插件。用户可以在聊天中粘贴物品文本，或输入物品名、通货名，插件返回实用的实时估价，并附带置信度、有效样本数、当前联盟等信息。

第一版优先保证准确性和可维护性，不追求功能堆叠。MVP 支持国际服繁体中文客户端复制出的完整物品文本查价、物品名/通货名快速查价、联盟配置。OCR、截图识别、剪贴板监听、价格提醒、走势图等功能不进入第一版。

## 语言约定

项目文档、设计说明、代码注释、测试说明、错误提示、用户可见输出默认使用中文。

以下内容保留英文或原始格式：

- Python 类名、函数名、变量名、配置键。
- API 路径、HTTP header、JSON 字段。
- POE2 物品文本中的原始繁体中文或英文词缀。
- 第三方服务、仓库、库名和协议名。

## 产品范围

### MVP 范围内

- AstrBot 插件骨架，包括 `metadata.yaml`、`main.py`、`_conf_schema.json`、`requirements.txt`。
- 聊天命令：
  - `/poe2查价 <物品文本或物品名>`
  - `/poe2price <物品文本或物品名>`
  - `/poe2联盟`
  - `/poe2联盟 <联盟名>`
  - `/poe2通货 <通货名>`
- 优先解析 POE2 国际服繁体中文客户端导出的物品文本，同时保留英文客户端文本的扩展能力。
- 通过 Path of Exile trade2 接口查询实时挂售。
- 通过 POE2 Scout 辅助查询通货和暗金的聚合价格。
- 处理限流、退避、缓存复用。
- 价格归一化、异常价格过滤、中位数/区间估算、置信度输出。
- 为解析器、查询构建器、估价器、格式化器编写单元测试。

### MVP 范围外

- OCR 或图片识别。
- Overlay 或任何直接游戏交互。
- 自动监听剪贴板、游戏日志或本地文件。
- 价格提醒和定时监控。
- 长期价格走势图图片。
- 简体中文、日文、韩文等其他语言的物品文本解析。
- OAuth 接入，除非后续 trade API 行为强制要求。

## 外部约束

### AstrBot 插件约束

- 插件入口文件必须是 `main.py`。
- 插件类必须继承 `Star`。
- 命令使用 AstrBot 的 `filter` 装饰器注册。
- 插件配置使用 `_conf_schema.json`。
- 网络请求使用异步库，例如 `httpx` 或 `aiohttp`，不使用 `requests`。
- 持久化数据存储到 AstrBot 数据目录，例如 `data/plugin_data/{plugin_name}/`，不写入插件源码目录。
- 插件卸载或停用时在 `terminate()` 中关闭异步 HTTP 客户端等资源。

### Path of Exile API 约束

- 请求必须带可识别、可配置的 `User-Agent`。
- 必须尊重 HTTP 429、`Retry-After` 和 rate-limit headers。
- 必须避免无效请求循环，因为连续 4xx 响应可能导致访问受限。
- GGG 官方开发者文档只承诺支持已公开列出的资源。trade2 搜索接口虽然被公开交易页面和多个第三方工具实际使用，但插件必须把它视为不稳定接口，并在失败时优雅降级。

## 推荐架构

采用 “Path of Exile trade2 实时挂售为主，POE2 Scout 聚合价格为辅” 的架构。

这样可以避免常见 overlay 的准确性问题：用历史均价或聚合均价给黄装估价。黄装价值高度依赖当前在线挂售和相似词缀，不能只看聚合价。

```text
main.py
services/
  item_parser.py
  item_text_normalizer.py
  stat_catalog.py
  trade_client.py
  poe2scout_client.py
  query_builder.py
  price_estimator.py
  formatter.py
models.py
```

## 模块设计

### `main.py`

职责：

- 注册 AstrBot 命令。
- 读取插件配置。
- 初始化各服务模块。
- 解析命令文本，并分发到对应查询流程。
- 通过 AstrBot 事件返回纯文本结果。
- 在 `terminate()` 中关闭异步客户端。

`main.py` 应保持轻量。业务逻辑应放到 `services/` 模块中。

### `models.py`

职责：

- 定义解析物品、词缀、挂售记录、估价结果、查询上下文、错误状态等数据结构。
- 为解析器、查询构建器、trade 客户端、估价器、格式化器提供稳定接口。

核心模型：

- `ParsedItem`
- `ItemModifier`
- `TradeListing`
- `PriceEstimate`
- `LookupContext`
- `RateLimitState`

### `services/item_parser.py`

职责：

- 将 POE2 国际服繁体中文物品文本解析成结构化数据，并为英文文本预留扩展入口。
- 识别物品类别、稀有度、名称、基底、物品等级、品质、孔位、宝石等级、腐化状态、武器 DPS、防御属性、词缀。
- 区分固有词缀、显式词缀、工艺词缀。
- 从词缀文本中提取数值。

解析器要能容忍不完整的粘贴文本。如果输入不是完整物品文本，返回 `None`，由命令处理层按物品名查询处理。

繁体中文文本必须支持以下字段和结构：

- `物品種類:` 对应 item class。
- `稀有度:` 对应 rarity。
- `普通`、`魔法`、`稀有`、`傳奇`、`通貨`、`寶石` 等稀有度文本应映射到内部标准枚举。
- `物品等級:` 对应 item level。
- `僅限:`、`需求:`、`品質:`、`插槽:`、`已汙染` 等常见字段应作为可选属性解析。
- `--------` 作为分段符。
- `{ 前綴 "" }`、`{ 後綴 "" }`、`褻瀆前綴`、`褻瀆後綴` 这类词缀标签和污染词缀标记要能识别，不能误判为物品名称或基底。
- 说明文字、风味文本、操作提示，例如“放置到一个天赋树的珠宝插槽中以产生效果”，应忽略，不参与查价条件。
- 对暗金/传奇物品，应正确识别第一段中的物品名和基底，例如 `水井之心` 是物品名，`鑽石` 是基底。

### `services/item_text_normalizer.py`

职责：

- 将繁体中文物品文本字段归一化为内部字段名。
- 维护语言别名字典，例如 `物品種類`、`Item Class` 都归一到 `item_class`。
- 维护稀有度映射，例如 `傳奇` 归一到 `unique`。
- 清理不会参与查询的展示文本、花括号标签、风味文本、操作提示。
- 保留原始物品名、原始基底、原始词缀文本，便于调试和用户输出。

该模块不直接访问网络，只负责纯文本标准化，方便单元测试覆盖。

### `services/stat_catalog.py`

职责：

- 拉取并缓存 `/api/trade2/data/stats`。
- 将 stat 文本标准化，把具体数值替换成占位符。
- 将解析出的繁体中文或英文词缀文本映射到官方 stat id。
- 将缓存持久化到 AstrBot 插件数据目录，并支持配置 TTL。

如果 stat 列表加载失败，黄装查价应返回清晰的“无法解析词缀”提示，不能输出误导性的精确价格。

繁体中文词缀映射策略：

- 优先使用 trade2 stats 接口返回的本地化 stat 文本进行匹配；如果接口只返回英文 stat 文本，则使用插件内置的繁中到标准 stat 模板映射。
- 第一版必须覆盖高价值常见词缀：生命、抗性、移动速度、技能等级、攻击速度、施法速度、暴击、物理/元素伤害、防御值。
- 对无法映射的词缀，不应强行加入查询；应记录为未匹配词缀，并在低置信度或调试信息中体现。

### `services/query_builder.py`

职责：

- 根据 `ParsedItem` 构建 trade2 查询 JSON。
- 构建物品名查询、通货查询、基础物品查询。
- 先构建严格查询，再生成逐步放宽的查询方案。

严格查询规则：

- 暗金：优先按物品名称和类型查询。
- 通货/基础物品：按名称或类型查询，并要求有标价。
- 黄装装备：按基底、稀有度、有标价、在线状态、关键数值过滤、高价值词缀 stat id 查询。

黄装放宽规则：

- 优先保留高价值词缀，例如最大生命、抗性、移动速度、技能等级、攻击速度、施法速度、暴击相关词缀、DPS 相关词缀。
- 先丢弃低价值词缀。
- 保守降低数值门槛。
- 当找到足够可比样本，或达到配置的最大放宽层级时停止。

繁体中文物品名查询规则：

- 如果 trade2 接口接受繁体中文 `name`/`type`，优先使用原始繁中名称和基底查询。
- 如果繁中名称无法得到结果，应尝试通过 POE2 Scout、trade data 或插件内置别名字典转换到英文或 API 可识别名称。
- 对无法转换名称的暗金，应返回“名称映射失败/无结果”的明确提示，不能凭基底或聚合价格输出精确估价。

### `services/trade_client.py`

职责：

- 使用异步 HTTP 客户端访问 `https://www.pathofexile.com/api/trade2`。
- 搜索接口：
  - `POST /api/trade2/search/poe2/{league}`
- 拉取详情接口：
  - `GET /api/trade2/fetch/{ids}?query={query_id}`
- 联盟接口：
  - `GET /api/trade2/data/leagues`
- stat 接口可由本模块提供，也可由 `stat_catalog.py` 内部调用。
- 解析限流 headers，并尊重 429 `Retry-After`。
- 批量拉取挂售详情，默认每批 10 个 id。
- 将 API 响应转换成 `TradeListing` 模型。

客户端必须实现一个小型熔断器。如果遇到限流或 403，应在短时间内停止继续请求 trade API，并返回受控错误。

### `services/poe2scout_client.py`

职责：

- 使用异步 HTTP 客户端访问 POE2 Scout 公共 API。
- 拉取 realm 和 league 信息。
- 拉取通货、暗金的聚合价格。
- 用较短 TTL 缓存响应。

POE2 Scout 是辅助数据源：

- `/poe2通货` 优先使用 POE2 Scout。
- 暗金查价可用 POE2 Scout 提供聚合价背景。
- 黄装不能只依赖 POE2 Scout 定价。

### `services/price_estimator.py`

职责：

- 将挂售价格归一化到配置的参考通货，POE2 联盟市场优先使用 exalted orb。
- 过滤无效挂售、未标价挂售、不支持的通货、明显异常价格。
- 使用中位数和类似四分位区间的方式估算价格，不使用简单平均值。
- 根据以下因素输出置信度：
  - 有效样本数。
  - 匹配词缀数量。
  - 查询放宽层级。
  - 价格离散程度。
  - 在线状态和数据新鲜度。

置信度规则：

- 高：样本足够、价格离散低、查询严格或仅轻微放宽。
- 中：样本足够，但价格离散较大或查询有中等放宽。
- 低：样本少、查询严重放宽、币种混杂或可比性弱。

如果有效样本少于配置下限，返回低置信度结果和明确警告，不输出看似精确的单点价格。

### `services/formatter.py`

职责：

- 将估价结果格式化为适合聊天发送的中文文本。
- 输出内容包括：
  - 物品显示名。
  - 联盟。
  - 估价区间。
  - 中位数。
  - 置信度。
  - 有效样本数。
  - 查询放宽说明。
  - 前几条参考挂售。
  - 限流、样本不足等警告。

输出文案必须避免过度断言。对于样本稀疏市场，应明确写“样本不足，仅供参考”。

## 配置设计

使用 `_conf_schema.json`，包含以下字段：

- `default_league`：字符串，默认 `Runes of Aldur`。
- `realm`：字符串，默认 `poe2`。
- `max_fetch_results`：整数，默认 `20`。
- `min_valid_listings`：整数，默认 `5`。
- `cache_ttl_seconds`：整数，默认 `86400`。
- `scout_cache_ttl_seconds`：整数，默认 `1800`。
- `request_timeout_seconds`：整数，默认 `15`。
- `enable_poe2scout`：布尔值，默认 `true`。
- `user_agent_contact`：字符串，默认空。为空时插件应提示管理员填写，因为负责任地使用 API 需要可识别的联系信息。
- `poesessid`：字符串，默认空。如果 AstrBot 配置面板支持隐藏字段，应设置为隐藏；仅在后续确实需要时使用。

## 命令流程

### 完整物品文本查价

1. 用户发送 `/poe2查价` 并附带完整物品文本。
2. `main.py` 判断输入是物品文本，并调用 `item_parser`。
3. `stat_catalog` 解析词缀对应的 stat id。
4. `query_builder` 构建严格查询和放宽查询。
5. `trade_client` 执行 search/fetch，直到获取足够可比挂售，或耗尽查询方案。
6. `price_estimator` 计算估价区间、中位数、置信度。
7. `formatter` 返回中文聊天结果。

### 物品名查询

1. 用户发送 `/poe2查价 Dream Fragments`。
2. 如果输入无法解析为完整物品文本，则按物品名或类型查询。
3. 对暗金和通货，先尝试 POE2 Scout 获取聚合价格背景。
4. 使用 trade2 查询当前实时挂售。
5. 如果可用，同时返回聚合价格背景和实时挂售估价。

### 通货查询

1. 用户发送 `/poe2通货 divine`。
2. 优先查询 POE2 Scout 当前联盟汇率。
3. 如果 POE2 Scout 不可用，回退到 trade2 查询。
4. 返回相对参考通货的价格，并标明数据来源。

### 联盟管理

1. `/poe2联盟` 返回当前默认联盟和可用 POE2 trade 联盟列表。
2. `/poe2联盟 <联盟名>` 将插件级默认联盟保存到 AstrBot 插件配置或插件 KV 存储。
3. 用户级联盟默认值不进入 MVP，后续按需求扩展。

## 错误处理

- 解析失败：说明未识别输入，并给出正确命令格式。
- stat id 匹配失败：仅在安全时退化为名称/基底查询，否则提示词缀无法匹配。
- 无挂售结果：提示当前没有可比挂售，并建议检查联盟或输入。
- 429：停止 trade 请求，并显示可重试等待时间。
- 403：提示当前 trade 访问被拒绝，并避免立即重试。
- 超时：返回临时失败提示。
- POE2 Scout 失败：继续尝试 trade2 实时挂售。
- 黄装 trade2 失败：不能把聚合价当成精确黄装估价返回。

## 测试策略

单元测试：

- 解析繁体中文 normal、magic、rare、unique、currency、gem 的代表性 POE2 物品文本。
- 覆盖用户提供的传奇珠宝样例：`物品種類: 珠寶`、`稀有度: 傳奇`、`水井之心`、`鑽石`、`物品等級: 81`、`褻瀆前綴`、`褻瀆後綴`。
- 验证风味文本和操作提示不会被解析成词缀或物品名称。
- 验证英文客户端文本作为兼容输入不会破坏解析架构。
- 用 fixture 测试词缀标准化和 stat id 解析。
- 用 fixture 测试繁体中文字段名和稀有度映射。
- 测试严格黄装查询和放宽黄装查询的 JSON 构建。
- 用包含异常价和混合币种的 fixture 测试估价逻辑。
- 测试高/中/低置信度、无结果、限流等输出文案。

集成式测试：

- Mock trade search、trade fetch、POE2 Scout、rate-limit headers。
- 验证上游失败时能复用缓存。

人工验收：

- 在本地 AstrBot 实例中运行命令。
- 验证 `/poe2联盟` 能返回当前 POE2 联盟。
- 粘贴一个繁体中文黄装，确认返回价格区间和置信度。
- 粘贴用户提供的 `水井之心` 传奇珠宝样例，确认能识别物品名、基底、稀有度和物品等级，并执行暗金查价流程。
- 查询一个繁体中文暗金名，确认能返回聚合价背景和实时挂售，或在名称映射失败时给出明确提示。
- 用 429 fixture 验证退避提示。

## 实现顺序

1. 创建插件骨架和配置 schema。
2. 添加繁体中文文本归一化器，并写字段映射测试。
3. 添加模型和解析器，并写繁中物品文本测试。
4. 添加 stat catalog 缓存和繁中词缀映射，并写测试。
5. 添加 query builder，并写测试。
6. 添加 trade client 和 mock HTTP 测试。
7. 添加 POE2 Scout client 和 mock HTTP 测试。
8. 添加 price estimator 和 formatter，并写测试。
9. 接入 AstrBot 命令。
10. 运行本地 lint 和测试。
11. 在 AstrBot 中人工验收。

## 已明确延后

- 用户级联盟设置延后，MVP 使用插件级默认联盟。
- OAuth 延后，除非后续 authenticated access 成为必要条件。
- OCR 和图片识别延后，第一版先保证文本查价准确可靠。
